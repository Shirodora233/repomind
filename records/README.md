# 项目记录目录

本目录用于记录项目推进过程，不作为最终交付文档目录。

`records` 主要放三类内容：

- 阶段实现记录：每个阶段一个文件，持续更新。
- 技术问题与解决方式：集中记录反复出现的问题、原因和处理方案，并在过时时更新、归档或删除。
- 实验过程备忘：在正式实验报告形成前，记录配置、问题和阶段性结果。

正式文档、数据集说明、评测方案说明、实验报告等应放在 `docs/` 中，并在对应工作真正实现或结果稳定后再写。

## 记录方式

实施记录按阶段维护，不要求每轮对话或每次小修改都新增文件。阶段内的小进展、问题和验证结果应追加到对应阶段文件中。

建议阶段包括：

- `01-project-initialization-and-documentation.md`：项目初始化与文档确立阶段。
- `02-test-case-construction.md`：构建测试样例阶段。
- `03-oracle-context-evaluation.md`：Oracle Context 测试阶段。
- `04-rag-agentic-retrieval.md`：RAG / Agentic Retrieval 阶段。
- `05-fine-tune-and-ablation.md`：Fine-tune 与消融实验阶段。

## 什么时候需要更新记录

以下情况应更新对应阶段记录：

- 设计或修改 baseline case。
- 实现或调整 prompt、RAG、agent loop、评分脚本、数据处理脚本。
- 跑模型实验或消融实验。
- 发现误报、漏报、检索失败、评分争议等问题。
- 修改评测协议、case schema、模型池或实验矩阵。
- 做出影响后续工作的设计决策。

以下情况才建议新增阶段文件：

- 进入新的主要工作阶段。
- 现有阶段文件无法自然承载新主题。
- 某个阶段过大，需要拆分成独立子阶段。

## 与 docs 的区别

`records` 记录“项目怎么推进、遇到了什么、怎么解决”。

`docs` 记录“已经形成的正式说明”，例如：

- 数据集 1.0 包含哪些 case。
- 数据集如何分层、如何标注。
- Oracle Context 测评怎么做。
- RAG / E2E 测评怎么做。
- 测试点、指标、评分规则和实验报告。

不要在 `docs` 中放零散过程记录，也不要提前创建大量未实现阶段的正式文档。

## 技术问题记录维护

技术问题记录不是永久真理。复用 `technical-issues-and-solutions.md` 中的解决方式前，应检查它是否仍符合当前代码、工具、环境和实验阶段。

如果已有问题记录不再适用，应：

- 更新状态和最后复核时间。
- 修正原因判断或解决方式。
- 对被新实现替代的记录标记为 `stale` 或 `archived`。
- 对明显会误导后续工作的记录进行删除或移入归档区。

## 2026-06-19 补充：reports 目录

正式实验报告不再放入 `records/` 或 `docs/`，统一放入 `reports/`。

- `runs/`：原始实验输出，不提交。
- `reports/`：正式 baseline、E2E、消融和模型对比报告，提交。
- `records/`：阶段进展、实施记录、实验过程备忘和问题摘要。
- `docs/`：稳定方法文档，例如数据集说明、评测协议、评分规则。

阶段记录中只保留实验完成状态、run path、report path 和关键结论摘要；完整指标、分 case 表格、成本和失败分析放入对应 report。
